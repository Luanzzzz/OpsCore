# Phase 2 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none |
| Icon library | lucide-react or equivalent linear icon set |
| Font | IBM Plex Sans |

**Direcao:** manual design system, web-first, preservando a linguagem operacional contida da Fase 1 enquanto expande o workspace de triagem para execucao rastreavel. A UI deve parecer uma superficie operacional continua, nao um novo modulo acoplado por cima do produto.

---

## Primary Screen Contract

| Element | Contract |
|---------|----------|
| Primary screen | Execucao de trabalho |
| Focal point | Lista densa de tarefas ativas com responsavel, prioridade, status e envelhecimento visiveis sem abrir o detalhe |
| Secondary anchor | Painel lateral com detalhe da tarefa, vinculo com a origem e historico operacional essencial |
| Tertiary zone | Faixa compacta de itens prontos para conversao e blocos-resumo de gargalos de execucao |
| Navigation model | Sidebar compacta com `Inbox` e `Tarefas`, header funcional e area principal em layout list-detail no desktop |
| Mobile fallback | Lista de tarefas em tela principal; detalhe e conversao abrem como tela dedicada ou painel de tela cheia |

**Visual hierarchy:**
1. Tarefas que exigem acao agora: `Critica`, `Sem responsavel`, `Bloqueada` ou envelhecidas
2. Estado de ownership e progresso na linha da tarefa
3. Vinculo com item de origem, contexto da triagem e proxima acao no detalhe

**Accessibility note:** acoes de conversao, ownership e status devem manter label textual visivel ou `aria-label` explicita. Chips e indicadores de estado nao podem depender apenas de cor.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label | 14px | 600 | 1.3 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px | 600 | 1.1 |

**Type rules:**
- Maximo de 4 tamanhos e 2 pesos.
- Titulos de tarefa usam `Body` com contraste forte; owner, origem e SLA usam `Label`.
- Contadores de backlog e itens sem responsavel podem usar `Display` apenas em blocos de resumo prioritarios.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | #F4F1EA | Background geral, canvas principal, areas de respiro |
| Secondary (30%) | #D9E2D3 | Sidebar, paineis secundarios, faixa de conversao e blocos de resumo |
| Accent (10%) | #C65D2E | CTA `Converter em tarefa`, item ativo da navegacao, foco de teclado, linha selecionada, contadores criticos |
| Destructive | #A63A2F | Acoes destrutivas, encerramento/arquivamento de tarefa, erros criticos |

Accent reserved for: `Converter em tarefa`, estado ativo da navegacao `Tarefas`, foco de teclado, destaque da tarefa selecionada, badge `Critica` e contadores de backlog realmente urgentes.

**Semantic support colors:**
- Stable/success: `#4F6D5A`
- Warning/waiting: `#B78A2A`
- Neutral text strong: `#243126`
- Neutral text soft: `#5A6B5F`

**Color behavior:**
- `Sem responsavel` usa warning com texto explicito, nunca accent.
- `Bloqueada` pode usar destructive para sinalizacao do estado, mas acoes destrutivas continuam reservadas a confirmacoes.
- O vinculo com a origem deve usar tons neutros ou secundarios, nunca competir com o CTA de conversao.

## Component and Layout Contract

| Area | Contract |
|------|----------|
| Task list | Estrutura dominante em lista densa ou tabela operacional; nao usar kanban como layout principal da fase |
| Conversion strip | Faixa compacta acima da lista principal com itens triados e prontos para virar tarefa |
| Task detail | Secoes claras para ownership, progresso, origem vinculada, contexto herdado da triagem e historico essencial |
| Conversion flow | Painel deslizante, modal largo ou painel lateral com dados da origem em leitura e campos editaveis da tarefa |
| Ownership controls | Responsavel sempre visivel como campo de primeira linha; tarefas sem responsavel devem se destacar antes de tarefas concluidas |
| Summary blocks | 4-5 blocos maximos: itens prontos para conversao, tarefas sem responsavel, tarefas bloqueadas, tarefas criticas, tarefas envelhecidas |
| Filters/sort | Barra funcional com filtros por responsavel, status, prioridade e envelhecimento visiveis sem menus profundos |
| Origin linkage | Card persistente no detalhe com titulo/origem/status do item original e acesso claro ao contexto herdado |

**Layout rule:** a fase 2 deve mostrar explicitamente a diferenca entre backlog de entrada e backlog de execucao. Isso pode aparecer via navegacao, faixa de conversao ou resumo contextual, mas a separacao nao pode ficar implícita.

## Interaction and States

| State | Contract |
|-------|----------|
| Default task list | Operador consegue varrer responsavel, status, prioridade e envelhecimento em uma leitura rapida |
| Ready-to-convert item | CTA `Converter em tarefa` aparece de forma clara quando a entrada ainda nao gerou trabalho operacional |
| Converted task | O detalhe mostra de imediato o vinculo com o item original e quem assumiu a execucao |
| Unassigned task | Destacada como pendencia operacional primaria, com copy e sinal visual claros |
| Loading | Skeletons discretos para lista, detalhe, faixa de conversao e blocos-resumo |
| Empty execution backlog | Explica que ainda nao ha tarefas e empurra o usuario para converter uma entrada triada |
| Error | Explica o problema e aponta para atualizar a tela ou revisar a ultima conversao |
| Status change | Mudanca de status deve refletir progresso real da execucao, nao apenas cosmetica na UI |

**Interaction priorities:**
- Clique principal seleciona tarefa e atualiza o detalhe sem tirar o contexto da lista.
- `Converter em tarefa` deve prefilar prioridade e contexto da origem, mas manter responsavel e status como decisao humana.
- Acoes rapidas de status ou responsavel podem existir, mas nunca devem esconder o contexto da origem.
- Ordenacao default privilegia prioridade e envelhecimento, com responsavel/status como filtros operacionais secundarios.

## Motion Contract

| Motion | Rule |
|--------|------|
| List/detail transition | 120-180ms, easing suave, sem overshoot |
| Fluxo de conversao | 160-200ms, entrada limpa, sem animacao ornamental |
| Atualizacoes de status e responsavel | Fade ou troca curta de cor/opacidade, sem microinteracao chamativa |
| Summary refresh | Atualizacao suave com fade leve ou count-up curto |

**Do not use:** colunas arrastaveis como foco principal, animacoes de board, parallax ou qualquer efeito que aproxime a experiencia de um app de produtividade generico.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Converter em tarefa |
| Empty state heading | Nenhuma tarefa em execucao ainda |
| Empty state body | Converta uma entrada triada para iniciar o backlog de execucao, definir responsavel e acompanhar o progresso sem perder a origem. |
| Error state | Nao foi possivel carregar o backlog de execucao. Atualize a tela ou revise a ultima conversao realizada. |
| Destructive confirmation | Encerrar tarefa: confirme somente se o trabalho realmente saiu do backlog ativo e o historico com a origem ja esta preservado. |

**Tone:** claro, operacional e direto. Evitar linguagem de ferramenta de projetos genérica, jargao corporativo vazio ou copy de help desk.

**Secondary labels recommended:**
- `Itens prontos para conversao`
- `Responsavel`
- `Status da tarefa`
- `Origem vinculada`
- `Sem responsavel`
- `Bloqueada`
- `Ultima movimentacao`
